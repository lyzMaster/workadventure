## 1. 执行摘要

1. **现状：WorkAdventure 很适合复用其 TMJ 渲染、人物 Sprite、Inline Map Editor、WAM Entity/Area 和多人位置同步，但不应直接把整个产品模型等同于 WAM。**
2. **现状：开发者基础地图是 TMJ；用户编辑数据主要落在 WAM 的 `entities`、`areas`、`settings`、`metadata` 中。**
3. **现状：浏览器经 WebSocket 连接 `play/pusher`，pusher 经双向 gRPC 连接 `back`；地图编辑命令由 `back` 经 gRPC 发往 `map-storage`。**
4. **现状：家具编辑采用乐观执行：前端先修改 Phaser/WAM 内存，再发送 Protobuf Command；服务端串行处理后广播。**
5. **现状：Map Storage 最终每 15 秒将完整 WAM `JSON.stringify` 写回文件，不是以 JSON Patch 作为 Inline Editor 的主存储格式。**
6. **现状：Map Storage 另有受认证的 HTTP `PATCH *.wam` JSON Patch API，但它不是当前实时编辑链路的一部分。**
7. **现状：WAM 已有 `version: "2.1.0"`、迁移器、Zod 校验和稳定的 Entity UUID；适合传输，但不宜直接成为长期 App/iCloud 存档协议。**
8. **现状：当前人物公共渲染基类是 `Character`，具体子类只有本地 `Player` 和网络绑定的 `RemotePlayer`；仓库没有独立 `NPC` 或 `AgentCharacter`。**
9. **部分具备：`Character` 已提供分层外观、方向、Walk/Idle 动画、名字、状态、气泡、路径跟随和完成 Promise，可以高度复用于 Agent 人物。**
10. **不存在：当前没有能够脱离真实 `userId`、RoomConnection 和用户消息而长期管理多个本地人物的运行时 Repository。**
11. **部分具备：现有 `PathfindingManager` 会避开碰撞，但一个实例只容许一个正在计算的路径；并发 Agent 寻路会互相取消。**
12. **现状：区域进入/离开及脚本属性目前由当前玩家位置驱动；远程人物和未来 Agent 人物不会自动触发同一套 Area 事件。**
13. **建议：Home 和 Office 应作为两个独立 TMJ/WAM 场景，不应只是同一地图中的两个 Area。**
14. **建议：新增独立 Command Gateway，复用 Map Editor Command、`Character`、路径和权限模块，但不把 Agent 暴露给 Phaser GameObject。**
15. **建议：iCloud 保存独立 `AppWorld` JSON；WAM Entity/Area 应转换为用户 Overlay，而不是保存整个服务器 WAM。**
16. **建议：Agent 人物应使用独立 `AgentCharacter` + `AgentCharacterRepository`，默认存在于 App/iCloud 数据，不伪装成 `RemotePlayer`。**
17. **建议：Agent 高层命令以 Entity/Area/Character 作为目标；坐标、帧和 Phaser 对象只应是控制器内部细节。**
18. **阻塞风险：`play`、`back`、`map-storage`、`uploader` 自带许可证为 AGPLv3 + Commons Clause；闭源收费 iOS 产品必须在实施前取得法律意见或商业许可。**
19. **高风险：当前前端虽能生成静态资源，但启动和进图流程依赖 `/map`、`/anonymLogin`、Woka API 和 Room WebSocket；离线 WKWebView 不是“直接打包即可”。**
20. **推荐最小切入点：先保留在线服务，完成两个独立 WAM 场景、单件家具持久化和一个本地 Agent 人物；验证模型后再抽离离线运行与 iCloud。**

### 4.6 并发和失败语义

**现状：**

### 5.1 TMJ 与 WAM 边界

- TMJ：开发者维护的基础地形、Tile Layer、碰撞 Tile、对象层、出生点和基础资源。
- WAM：可编辑 Overlay、Area、Entity Collection 引用、地图设置和元数据。
- WAM 用 `mapUrl` 引用 TMJ，而不是内嵌 TMJ。
- 用户家具最终存放在 WAM `entities: Record<string,WAMEntityData>`。
- Area 存放在 WAM `areas: AreaData[]`。
- Entity ID 通常由前端 UUID v4 生成。
- Entity 素材通过 `{collectionName,id}` 的 `prefabRef` 引用。
- Map Storage 持久化完整 WAM；命令队列只保留约 30 秒用于追赶。

### 5.2 版本、未知字段和迁移

- WAM 当前最新版本为 `2.1.0`。  
  [WamFileMigration.ts](/Users/bytedance/Documents/myprojects/workadventure/libs/map-editor/src/Migrations/WamFileMigration.ts:13)
- 已有 `1.0.0 → 2.0.0 → 2.1.0` 迁移。
- Entity Collection 最新 `1.0`，旧版 ID 由 collection/name/color/direction 生成。  
  [EntitiesFileMigration.ts](/Users/bytedance/Documents/myprojects/workadventure/libs/map-editor/src/Migrations/EntitiesFileMigration.ts:15)
- Zod Object 默认会剔除未声明字段；因此普通未知字段不应依赖保留。
- `vendor: unknown` 是明确的扩展逃生口。
- `PropertyBase.serverData: unknown` 可含服务器生成数据；不应未经筛选写入个人 iCloud。
- `lastCommandId` 是实时追赶游标，不是业务版本或乐观锁版本。

不能直接保存到 iCloud 的典型对象：

- `GameMap`、`WamFile`、`Entity`、`Character`、`Player`、`RemotePlayer`
- `Map`、`Set`、RxJS Subject、Svelte Store

## 6. Home / Office 场景模型

### 6.1 当前场景切换模型

- 场景标识主要是 `Room.key`，来源于规范化 URL path。
- `Room.group` 对应 SaaS 中的 world 概念。
- 后端 `GameRoom` 以 room URL 维护用户和变量。
- Map Storage 以完整 WAM URL 映射为 `mapKey`。
- `WAM.mapUrl` 将一个 WAM 绑定到一个基础 TMJ。

切换时：

1. 出口解析为目标 URL。
2. `Room.createRoom` 取得目标 `mapUrl/wamUrl`。
3. 预加载目标 `GameScene`。
4. 关闭旧 Room WebSocket 和旧 Scene Manager。
5. 新 Scene 重新加入后端房间。
6. 新出生位置由 URL hash/Area/Layer/Tile 计算。
7. 全局用户身份、LocalStorage、Svelte 应用壳继续存在；Scene 级 Manager 和 Store 被重建。

### 6.2 方案比较

| 维度 | A：Home/Office 两个 TMJ/WAM | B：同一地图两个 Area |
|---|---|---|
| 地图加载 | 各自独立资源和出生点，符合现有 Room/Scene | 初始加载更大，无法独立卸载 |
| 多人房间 | 天然是两个 GameRoom | 所有人仍在同一 GameRoom |
| 数据隔离 | 两个 WAM、两个 `mapKey` | Entity/Area 混在同一 WAM |
| iCloud | `scenes.home`、`scenes.office` 直接隔离 | 必须给每条用户数据额外打 Area 归属 |
| Agent 控制 | Scene 切换有明确边界和恢复点 | Agent 可能在未渲染区域继续运行，边界模糊 |
| 后续扩展 | 可独立升级、分享、多人权限 | 地图越来越大，版本/权限耦合 |

**建议：选择方案 A。**

推荐稳定标识不要直接使用 URL：

```text
sceneId: "home" | "office"
baseMapId: 产品稳定 ID
baseMapRevision: 开发者基础图版本
roomUrl: 当前 WorkAdventure 路由
wamUrl: 在线模式的存储地址
```

同一用户的装修数据以 `sceneId` 隔离；在线 WAM URL 和 iCloud Scene ID 建映射，不把环境域名当长期主键。

### 10.3 WKWebView 影响

**现状：**

- Vite 可输出 `play/dist/public` 静态资源。  
  [vite.config.ts](/Users/bytedance/Documents/myprojects/workadventure/play/vite.config.ts:35)
- Phaser 本身是浏览器 Canvas/WebGL 运行时，源码没有依赖桌面专有 API。
- TMJ、Tileset、Collection、图片和音频大量通过 `new URL(relative, base)` 解析，适合 HTTP 相对 URL。
- 当前启动流程仍请求 pusher `/map`、`/anonymLogin`、Woka 列表并连接 `/ws/room`。
- Service Worker 使用根路径 `/service-worker-*.js`，`file://` 下注册会失败。
- 地图脚本和网页大量依赖 iframe、origin、CORS、CSP、`postMessage`。
- 媒体依赖 `getUserMedia`、WebRTC、权限和安全上下文。
- `window.location.pathname` 被当成 Room ID；`file://` 不符合现有路由假设。

结论：

- 在线多人模式：用 WKWebView 加载 HTTPS 部署的 WorkAdventure 最接近现状。
- 离线单人模式：建议内嵌本地 HTTP Server，或新增离线 `RoomResolver/RoomConnection` 适配层；不建议直接依赖 `file://`。
- Swift Bridge 最适合调用 Command Gateway，而不是现有 iframe API 或 Phaser Scene。
- 可以只向 `window` 暴露一个窄接口，如 `window.appWorld.execute(command)`；内部仍做 Zod 校验、权限和碰撞检查。
- iOS 后台恢复需重建 Scene、Texture、Physics、路径、媒体和 Bridge pending request；只从 AppWorld JSON 恢复逻辑状态。


### 12.2 WAM 是否适合作为 iCloud 格式

**适合作为转换来源，不适合作为最终 App 协议。**

原因：

- WAM 同时包含开发者地图引用和用户 Overlay。
- `settings` 包含会议、录制等 WorkAdventure 平台语义。
- `metadata`/`vendor`/`serverData` 可能混入管理数据。
- `lastCommandId` 是在线编辑实现细节。
- Area Property 可含 Matrix Room ID、外部 URL、权限 tag。
- 没有 scene ownership、用户数据版本、基础地图升级迁移记录。
- Agent 人物无法自然表达为现有 Entity。

建议采用独立 `AppWorld`：

```text
AppWorld
├── schemaVersion
├── activeSceneId
├── scenes
│   ├── home
│   │   ├── baseMapId
│   │   ├── baseMapRevision
│   │   ├── furnitureOverlay
│   │   ├── customAreas
│   │   └── sceneSettings
│   └── office
├── characters
├── preferences
└── baseMapRevisions
```

### 12.3 转换规则

可直接转换：

- `WAMEntityData.x/y/name/prefabRef/properties`
- `AreaData` 的几何和安全 Property 子集
- 用户外观的 texture `id`，必要时附可迁移资源 key
- Entity UUID

需要筛选/转换：

- URL：转换为 bundle resource ID 或经过允许列表的远端 URL。
- `serverData`：默认不导出。
- `vendor`：只导出 App 自己 namespace 下的数据。
- Matrix/Jitsi/LiveKit 属性：离线存档通常剔除。
- `lastCommandId`：不导出。
- `ownerId`、权限 tag：按产品账户模型转换。
- `prefabRef`：要保存 Collection revision，避免基础素材升级后 ID 指向变化。

不应导出：

- auth token、Room token、Matrix token
- 当前 Remote Player
- WebSocket/Room/Group 信息
- Character/Entity Phaser 实例
- 当前动画帧和插值
- 临时路径、pending Command
- 浏览器媒体权限状态

### 12.4 基础地图升级

建议每个用户对象保留：

```text
entityId
baseMapRevisionAtCreation
prefabRef
position
optional anchorRef
```

升级流程：

1. 读取新基础 TMJ/WAM。
2. 校验 Prefab/Area/Anchor 是否仍存在。
3. 在临时副本上迁移用户 Overlay。
4. 重新跑边界和碰撞校验。
5. 产生迁移报告：保留、移动、缺失素材、无法放置。
6. 用户确认或自动备份后原子替换存档。

导入损坏 JSON 时：

- 先校验 `schemaVersion` 和大小；
- 迁移到最新 AppWorld；
- 在内存临时世界恢复；
- 每个 Scene 独立校验；
- 不覆盖最后正常存档；
- 保存上一个成功版本；
- 未知字段仅在明确 extension namespace 中保留。

### 12.5 Agent 人物持久化

Agent 不适合作为 WAM Entity，因为：

- Entity 的 Prefab、碰撞和 properties 是家具语义。
- Character 需要外观层、方向、动作和身份状态。
- WAM Entity 没有 Character 生命周期或动作协议。

建议 Agent 配置存 AppWorld：

```text
characterId
displayName
appearance
sceneId
defaultPosition
defaultDirection
state
behaviorRef
```

“当前位置”是否保存取决于产品语义：

- Home/Office 常驻角色：保存最后稳定位置或显式 home position。
- 临时对话动作：不保存中间路径和动画。
- 跨地图移动：完成 Scene 切换后再原子更新 `sceneId + position`。

## 13. 最小垂直切片计划

### 阶段 0：基线确认

- 修改范围：无产品修改；运行现有 build/typecheck/E2E，记录服务拓扑。
- 依赖：play、back、map-storage、至少一张可编辑 WAM。
- 验收：当前提交可进入地图、多人连接、编辑一件 Entity。
- 风险：本轮未执行完整构建/测试，基线状态仍需实际确认。
- 多人影响：无。
- 回滚：无变化。

### 阶段 1：Home / Office

- 修改范围：两张独立 TMJ/WAM、稳定 `sceneId ↔ roomUrl/wamUrl` 映射。
- 依赖：`Room`、`GameManager`、`StartPositionCalculator`。
- 验收：Home→Office→Home，出生点和 Room 切换正确。
- 风险：Scene cleanup 全量关闭媒体/iframe/Store，Host 状态必须放在 Scene 外。
- 多人影响：自然形成两个 GameRoom。
- 回滚：删除场景映射即可。

### 阶段 2：用户家具

- 修改范围：只复用现有 Inline Editor 和 Entity Command。
- 依赖：WAM、Map Storage、权限、Collection。
- 验收：放置沙发、移动/改方向/删除、刷新恢复；Home/Office 互不污染。
- 风险：失败回滚和并发语义不足。
- 多人影响：保留现有广播。
- 回滚：关闭 `ENABLE_MAP_EDITOR` 或移除用户 Overlay。

### 阶段 3：本地 Agent 人物

- 修改范围：`AgentCharacter`、Repository、Scene 生命周期 Adapter。
- 依赖：`Character`、外观 Loader、碰撞、路径。
- 验收：一个无 `userId`、无连接的 Agent 可 spawn、walk、stop、speak、remove。
- 风险：共享 PathfindingManager 会取消并发请求。
- 多人影响：本阶段不广播。
- 回滚：Repository 与 Scene Adapter 可整体关闭。

### 阶段 4：Command Gateway

- 修改范围：纯命令类型、Validator、Gateway、Furniture/Character/Scene Adapter。
- 依赖：Map Editor Command、Agent Repository、GameManager。
- 验收：Console 执行结构化命令；结果、取消、超时、完成事件可观察。
- 风险：若 Gateway 直接泄漏 Scene/GameObject，后续 WKWebView 会再次重构。
- 多人影响：家具沿原链；Agent 仍本地。
- 回滚：保留原 UI 路径，关闭 Gateway 暴露。

### 阶段 5：纯 JSON 导出

- 修改范围：AppWorld DTO、导入导出、迁移、校验和原子恢复。
- 依赖：Scene Registry、家具 Overlay、Agent 配置。
- 验收：导出→清空→导入，Home/Office/家具/人物一致；损坏文件不覆盖旧存档。
- 风险：基础地图 revision 和 Prefab ID 迁移。
- 多人影响：尚不处理在线冲突。
- 回滚：保留旧存档和导出版本。

### 阶段 6：iOS Bridge

- 修改范围：窄 `window.appWorld` Host Bridge、Swift 消息封装、本地 HTTP/bundle loader。
- 依赖：Command Gateway、AppWorld。
- 验收：Swift 加载/保存、发命令、收到完成事件；后台恢复可重建。
- 风险：WKWebView origin、Service Worker、媒体、iframe、后台挂起。
- 多人影响：在线模式继续用现有 RoomConnection。
- 回滚：Web 端 Gateway 不依赖 iOS，可独立运行。

### 风险分级

**阻塞风险**

- `play/back/map-storage/uploader` 的 AGPLv3 + Commons Clause 与闭源收费 iOS 产品的兼容性。  
  许可证明确称 Enterprise 可单独商业授权，并禁止“Sell the Software”定义下的收费产品/服务。  
  文件：[play/LICENSE.txt](/Users/bytedance/Documents/myprojects/workadventure/play/LICENSE.txt:679)  
  这不是法律意见，实施前需专业法律确认。
- 当前没有真正离线的无后端启动路径；纯 WKWebView bundle 需要架构适配。

**高风险**

- Agent 人物与真实 `RemotePlayer` 身份模型冲突。
- 多 Agent 并发寻路。
- Map Editor 乐观执行的失败回滚不足。
- 多人同时编辑同一 Entity 没有版本/冲突协议。
- 基础地图升级后的用户 Overlay 迁移。
- iCloud 与在线 WAM 双向修改冲突。
- WAM 被当作长期 App 协议会固化平台会议/权限语义。

**中风险**

- Phaser 当前依赖版本为 `4.2.0`；仓库对 Scene、Tilemap、Camera、Arcade Body 和 DOMElement 使用很深，升级/移动端渲染需独立基准。
- 移动端触摸、预览、拖拽、右键/Shift 网格交互与桌面语义不同。
- WKWebView 对媒体权限、后台恢复、第三方 Cookie、跨域 iframe 的限制。
- 多 Agent 网络同步带宽和 Viewport 过滤模型。
- Area 事件只绑定当前 Player。
- Prefab 外部 URL、素材许可和隐私 URL。

**低风险**

- Home/Office 两个独立 WAM 的数据隔离。
- Character Sprite/Walk/Idle/气泡复用。
- 关闭 Matrix、OIDC、Jitsi、LiveKit、上传和分析。
- 使用独立 AppWorld 保存纯 DTO。

地图及素材许可不能从主代码许可证推导。`WAMMetadata.copyright` 明确提示 Tileset/图片可有独立版权；仓库也包含素材级 LICENSE，例如 [megaphone/LICENSE.md](/Users/bytedance/Documents/myprojects/workadventure/play/public/resources/objects/megaphone/LICENSE.md)。发布前必须逐项盘点 bundle 中实际使用的地图、字体、Tileset 和图片。
